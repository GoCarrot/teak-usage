= Reward bundles
:page-aliases: usage:rewards.adoc

A reward bundle is a reusable reward configuration you attach to notifications and links. When a player clicks, Teak delivers the reward automatically — no extra code required after the initial integration.

Bundles live in *Rewards → Reward Bundles*. Create one once and attach it to as many notifications and links as you like.

== Set up items and currency

Your engineering team defines items and currency in code. Once they ship an update with those definitions, you can reference them in bundles.

=== Add an item or currency

. Select *Rewards* from the left rail
. Go to the *Items & Currencies* tab
. Click *New Item or Currency*
. Enter a display name and the Internal ID — this must match the code name exactly

NOTE: Display names are for your reference only and don't appear in notifications or link copy.

[[bundle-creation]]
== Create a bundle

You can create a bundle from the Rewards page, or directly from the notification editor while setting up a notification — a useful shortcut when you're building both at the same time.

=== From the Rewards page

. Select *Rewards* from the left rail
. Click *New Reward Bundle*
. Enter a name
. Add at least one reward: select the type and enter the amount
. Set *Reward expires* — how many days players have to claim after receiving the notification
. Click *Save*

=== From the notification editor

Open or create a notification and find the *Reward Bundle* field. Click it to create a new bundle inline. The same editor opens, and the bundle is saved to your library when you save.

[[randomized]]
== Add randomized rewards

The same reward every day gets predictable. Check *Use Random Rewards* to offer a drop table of possibilities — you can range from a modest base reward up to a jackpot, while keeping your average payout exactly where you want it.

Teak picks one reward option per claim, selected at random based on weighted chances.

. Check *Use Random Rewards* in the editor
. Add each reward option with its amount
. Set the *Chance* percentage for each option — all chances in a tier must add up to 100%

.Bundle editor with Use Reward Tiers and Use Random Rewards both enabled. Tier 1 is assigned to a specific audience with three randomized options. The Default Tier applies to everyone else.
image::rewards/2_bundle_editor_randomized.jpg[]

[TIP]
.Average claim amount
====
Teak calculates the average amount each claim is worth. This is the number to watch for economy balancing — a bundle ranging from 1M to 100M coins can have the same average payout as a fixed 5M coin bundle, letting you add excitement without changing what you actually spend on rewards.
====

[TIP]
.Personalization
====
When using randomized rewards, you can template the minimum or maximum reward amount into notification or email text using a xref:page$custom-tags.adoc[custom tag].
====

NOTE: When *Use Random Rewards* is on, Teak picks one reward option per claim. When it's off, players receive every item in the bundle — all reward types and all amounts.

[[tiered]]
== Add tiered rewards

For high-spend or deep-progression players, a reward that excites a casual player might barely register. Tiered rewarding lets you send one notification to a broad audience and still give each player an offer that's meaningful at their level — without creating separate notifications for each segment.

Check *Use Reward Tiers* to assign different rewards to different audiences.

Teak evaluates tiers from top to bottom. The first tier whose audience includes the player applies — that player receives only that tier's reward. The default tier catches anyone who doesn't match.

=== Set up tiers

. Check *Use Reward Tiers* in the editor
. Click *Insert Tier* to add a custom tier above the default
. Select an audience for the tier
. Add the rewards players in that audience should receive
. Use the reorder buttons to place your most specific audiences at the top — if a broad audience sits above a narrow one, players who qualify for the narrower tier will match the broader one first and never receive the higher-value reward
. Repeat for each additional tier

For a full walkthrough of tier evaluation, evaluation timing, and template tag integration, see xref:page$tiered-rewards.adoc[Tiered Rewards].

=== Combine tiers and randomized rewards

[TIP]
.Both features can be active on the same bundle
====
Enable *Use Reward Tiers* and *Use Random Rewards* together. Teak evaluates tiers first to select the reward set for that player, then randomizes within that set. Each tier can have its own reward options and independent odds — useful for giving VIPs better minimum and maximum rewards than everyone else.
====

== Test a bundle

Test a bundle to verify the right tier resolves and the reward payload is correct before attaching it to a live notification.

. Find the bundle and click *Edit ▾ → Test*
. Under *Test Reward for Audience*, select the tier to simulate
. Enter the *User ID* — the game-assigned ID of the player who should receive the reward
. Set *Posting User ID* to `0` for app-to-player notifications, or to the sharing player's ID for social rewards
. Select any applicable Extra Flags (`is_mobile`, `is_unity_webgl`)
. Click *Test*

Teak sends a real POST request to your reward endpoint. The dialog shows the full request and your server's response so you can confirm the integration is working.

== Duplicate a bundle

. Click *Edit ▾ → Duplicate*

Teak creates a copy with a new name and opens it in the editor. The copy is independent — changes to it don't affect the original. Duplicating is a quick way to build a tiered variant of an existing flat bundle without starting from scratch.

== Archive a bundle

Archiving removes a bundle from the active list without deleting it. Archived bundles can be restored at any time — useful for seasonal bundles you'll reuse later.

*To archive:* click *Edit ▾ → Archive*.

*To restore:* open the *Filter* dropdown, enable *Show archived*, find the bundle, and click *Restore*.

NOTE: If a bundle is attached to an active notification or link, Teak won't delete it — it will offer to archive it instead. This preserves the reward configuration for reference.

For help finding archived bundles and filtering the bundle list, see xref:page$bundle-groups-filtering.adoc[Bundle list filters].

[[audit-log]]
== Verify reward delivery

The Audit Log shows the outcome of every reward request Teak sent to your server.

. Select *Rewards* on the left rail
. Choose the *Audit Log* tab
. In the *Clicking User ID* box, enter the player's unique identifier
. Use the *Status* filter to narrow to Successes or Failures
. Set the date range and search

Results show:

* *Time:* when the player initiated receipt of the reward
* *Identifier:* the name of the notification or link clicked
* *Reward:* the item or currency and amount
* *Server Response:* the success or failure response from your server
